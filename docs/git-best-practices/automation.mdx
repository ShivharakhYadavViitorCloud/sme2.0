# Automation

Automate Git best practices to enforce them consistently across your team.

## 1. Branching Strategy

Automation can enforce branch naming conventions and structure:

### Branch Name Validation

Block PRs if branch names don't match defined patterns:

```
✅ feature/*
✅ bugfix/*
✅ hotfix/*
✅ docs/*
✅ chore/*

❌ update-stuff
❌ fix-bug
```

### Tools
- **commit lint** - Validate branch names via git hooks
- **branch-name-check** - GitHub Action for branch validation
- **CI Lint Rules** - Custom CI pipeline checks

## 2. Conventional Commits

Automation can enforce and validate commit messages:

### Reject Non-Standard Commits

```
❌ update stuff
❌ fix bug
❌ WIP

✅ feat(auth): add token refresh
✅ fix(login): handle invalid session
✅ docs(api): update endpoint documentation
```

### Tools
- **commitlint** - Validate commit messages against a config
- **husky** - Git hooks framework for pre-commit checks
- **CI Lint Rules** - Enforce in the CI pipeline

### Example commitlint Configuration

```javascript
// .commitlintrc.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [
      2,
      'always',
      ['feat', 'fix', 'refactor', 'docs', 'test', 'chore', 'perf', 'ci']
    ],
    'subject-case': [2, 'never', ['start-case', 'pascal-case', 'upper-case']],
    'subject-empty': [2, 'never'],
    'subject-full-stop': [2, 'never', '.'],
  }
};
```

## 3. Code Reviews

Automation can enforce and assist with code reviews:

### AI-Based Code Review

- Security vulnerability detection
- Performance issue flagging

### Human Review Enforcement

### Tools

## 4. Tag Releases Properly

Release tagging can be **fully automated via CI/CD**:

### Auto-Create Tags on Merge

Automatically generate semantic version tags when code merges to `main`:

```
main branch merge → analyze commits → create vX.Y.Z tag
```

### Tools
- **semantic-release** - Fully automated versioning and releases
- **GitHub Actions** - Custom release automation
- **standard-version** - Conventional commits to version management
- **Release Pipelines** - Azure Pipelines or GitLab CI

### Example GitHub Action

```yaml
name: Release
on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## 5. Maintain a CHANGELOG

CHANGELOG can be **fully auto-generated from commits**:

### Automated Generation

Parse conventional commits to generate changelog entries:

```
feat → Added
fix → Fixed
refactor → Changed
docs → Documentation updates
```

### Tools
- **semantic-release** - Auto-generates CHANGELOG.md
- **standard-version** - Generate CHANGELOG from commits
- **release-please** - Automated release PR creator
- **conventional-changelog** - Changelog generator

### Example Output

```markdown
## [1.2.0] - 2024-03-15

### Added
- User profile API endpoint
- Dark mode support

### Fixed
- Memory leak in WebSocket connection
- Session timeout handling

### Changed
- Updated authentication to OAuth 2.0
```

## 6. Secrets Must Never Enter Git

**Secrets can be automatically detected and prevented**:

### Automated Scanning

Tools scan commits for:
- AWS keys and credentials
- API tokens and authentication secrets
- Passwords and database URLs
- Private encryption keys
- OAuth tokens

### Tools
- **GitHub Secret Scanning** - Native GitHub scanning
- **GitLeaks** - Detect secrets in git history
- **TruffleHog** - Finds secrets in git repositories
- **Snyk** - Dependency vulnerabilities and secret detection

### Pre-Commit Protection

```bash
# .husky/pre-commit
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

# Scan for secrets before commit
npx detect-secrets scan --baseline .secrets.baseline
```

## 7. Maintain Repository Hygiene

Partially automated but highly beneficial:

### Automated Checks

- **Lint repository structure** - Ensure required files exist
- **Check for required files** - README, CONTRIBUTING, LICENSE
- **Dependency updates** - Automated security updates
- **CI checks** - Validate repository state on every push

### Tools
- **Dependabot** - Automated dependency updates
- **Renovate** - Dependency automation and updater
- **Pre-commit hooks** - Local repository validation
- **CI Repository Lint** - Custom CI checks

### Example Required Files Check

```yaml
# .github/workflows/repo-hygiene.yml
name: Repository Hygiene
on: [push, pull_request]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Check required files
        run: |
          test -f README.md || (echo "README.md missing" && exit 1)
          test -f CONTRIBUTING.md || (echo "CONTRIBUTING.md missing" && exit 1)
          test -f LICENSE || (echo "LICENSE missing" && exit 1)
```

## Practical Automation Summary

| Practice | Automation Level | Tools |
|----------|------------------|-------|
| Branching strategy | Fully enforceable | commitlint, GitHub Actions, branch protection rules |
| Short-lived branches | Partial | CI stale branch detection, automated cleanup |
| Conventional commits | Fully | commitlint, husky, CI validation |
| Mandatory reviews | Fully enforceable | GitHub branch protection, CODEOWNERS |
| Tag releases | Fully | semantic-release, standard-version |
| CHANGELOG | Fully auto-generated | semantic-release, release-please |
| Secrets scanning | Fully | GitHub Secret Scanning, GitLeaks, TruffleHog |
| Issue linking | Fully | GitHub Actions, PR templates |
| Repository hygiene | Partial | Dependabot, Renovate, CI checks |
| Clean history | Fully | Squash merge settings, rebase enforcement |
| Automation pipelines | Fully | CI/CD platforms (GitHub Actions, GitLab CI, etc.) |

## Getting Started with Automation

### Step 1: Choose Your Tools
Select tools based on your platform (GitHub, GitLab, Bitbucket) and team needs.

### Step 2: Configure Gradually
Start with one or two automations:
1. Conventional commits validation
2. Branch protection and required reviews

### Step 3: Expand Incrementally
Add more automations as the team adapts:
- Automated testing
- Secret scanning
- Dependency updates
- Release automation

### Step 4: Monitor and Adjust
Collect feedback and refine automation rules to match your team's workflow.

---

**Remember:** Automation should enable your team, not frustrate them. Start simple and iterate.
