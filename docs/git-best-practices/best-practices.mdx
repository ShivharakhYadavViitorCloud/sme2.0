# Git Best Practices

Proven strategies to maintain clean, organized, and productive Git workflows across your team.

## Branching Strategy (Avoid Chaos Early)

Adopt a consistent branching strategy across all repositories. Two popular approaches:

### GitFlow
Suitable for structured releases with multiple branch types:
- `main` - Production-ready code
- `develop` - Integration branch for features
- `feature/*` - Feature branches
- `release/*` - Release preparation branches
- `hotfix/*` - Emergency production fixes

### Trunk Based Development
Short-lived feature branches merged frequently into `main`:
- Reduces merge conflicts
- Enables continuous deployment
- Requires strong CI/CD pipeline

**Best Practice:** Choose one strategy and enforce it consistently across all repositories.

## Keep Branches Short-Lived

- Feature branches should ideally live **< 2–3 days**
- Merge or rebase frequently with the base branch to avoid large conflicts
- This reduces integration issues and simplifies code reviews

## Conventional Commits

Follow a standard commit format for clarity and automation:

```
type(scope): short summary
```

### Common Types
- `feat` - New feature
- `fix` - Bug fix
- `refactor` - Code refactoring
- `docs` - Documentation updates
- `test` - Test additions or changes
- `chore` - Maintenance tasks
- `perf` - Performance improvements
- `ci` - CI/CD configuration

### Examples
```
feat(auth): add token refresh endpoint
fix(login): handle invalid session gracefully
refactor(api): extract validation logic
docs(readme): update setup instructions
```

**Best Practice:** Keep commits small and meaningful. Each commit should represent a single logical change.

## Mandatory Code Reviews (Protected Branch)

- All changes must go through a Pull Request
- Ensure CI checks pass before approval
- Configure branch protection rules to prevent direct commits to `main`

### Benefits
- Catches bugs early
- Shares knowledge across the team
- Maintains code quality standards
- Provides audit trail for compliance

## Tag Releases Properly

Use semantic versioning for all production releases:

```
vMAJOR.MINOR.PATCH
```

### Examples
- `v1.0.0` - Initial release
- `v1.1.0` - New features added (backward compatible)
- `v1.1.1` - Bug fix (backward compatible)
- `v2.0.0` - Breaking changes

### Benefits
- Clear version traceability
- Easy rollback capability
- Dependency management for other projects
- Release history documentation

## Maintain a CHANGELOG

Document all release changes in `CHANGELOG.md`:

### Format
Categorize entries under:
- **Added** - New features introduced
- **Changed** - Changed functionality
- **Fixed** - Bug fixes
- **Removed** - Removed features
- **Deprecated** - Soon-to-be-removed features

### Example
```markdown
## [1.1.0] - 2024-01-15
### Added
- New API endpoint for user profiles
- Dark mode support

### Fixed
- Memory leak in websocket handler

### Changed
- Updated authentication flow to use OAuth 2.0
```

## Secrets Must Never Enter Git

Critical security principle:

- ❌ Do not commit credentials, tokens, or API keys
- ❌ Never hardcode passwords or database URLs
- ✅ Use environment variables or secret management tools
- ✅ Use `.env` files with `.gitignore` protection
- ✅ Implement secret scanning in CI pipeline

### Common Secrets to Protect
- Database credentials
- API keys and tokens
- AWS/Cloud credentials
- Private encryption keys
- OAuth secrets

## Use Issues + PR Linking

Connect work items with code changes:

### Examples
```
Fixes #123
Closes PROJ-456
Resolves #789, #790
```

### Benefits
- Automatic issue closing on merge
- Traceability from code to requirements
- Better project management
- Audit trail for compliance

## Maintain Repository Hygiene

Keep your repository clean and organized:

### Essential Files
- `README.md` - Project overview and setup instructions
- `CONTRIBUTING.md` - Contribution guidelines
- `CHANGELOG.md` - Release notes and changes
- `.gitignore` - Files to exclude from version control
- `LICENSE` - Project license

### Practices
- Remove unnecessary files and binaries
- Keep node_modules and build artifacts out of Git
- Archive old or deprecated branches
- Update documentation regularly
- Review and update dependencies

## Squash Merges or Rebase for Clean History

Maintain a clean, readable commit history:

### Squash Merge
Combines all commits into a single commit:
```bash
git merge --squash feature-branch
```
- Pros: Clean history, single commit per feature
- Cons: Loses individual commit details

### Rebase
Replays commits on top of the base branch:
```bash
git rebase main
```
- Pros: Linear history, preserves individual commits
- Cons: Requires force push (on your own branches only)

**Best Practice:** Choose one strategy for your team and enforce it consistently.

## Automate Everything Possible

Enable automation across your workflow:

- **CI Builds** - Automatic compilation and artifact generation
- **Tests** - Run test suite on every commit
- **Lint Checks** - Enforce code style automatically
- **Security Scans** - Detect vulnerabilities and secrets
- **Dependency Updates** - Automated dependency management
- **Release Automation** - Auto-generate versions and tags
